# 第三章 执行感知的多提议者BFT共识协议

## 3.1 问题描述 

共识协议的排序能力直接影响区块链的吞吐量和终局性，因此本章旨在设计一种面向稳定币高频转账场景的高性能BFT共识协议。根据排序架构的不同，区块链共识协议主要分为单提议者和多提议者两种类型。

在单提议者架构中，每轮共识仅由一个领导者节点负责打包交易并提议区块。经典的BFT协议如PBFT\[6\]、PoV与HotStuff\[3\]均属于此类。其中，PBFT通过两阶段投票实现确定性终局，但其通信复杂度为$O(n^{2})$，限制了可扩展性。PoV和HotStuff先后通过引入链式法定人数证书（Quorum Certificate，QC）和流水线设计，将通信复杂度降低至$O(n)$，并提供了简洁的安全性证明框架。然而，单提议者架构存在固有的性能瓶颈：领导者节点需要独自完成交易收集、批次构造和区块广播等任务，成为系统吞吐量的单点瓶颈。当网络规模增大或交易负载增高时，单个领导者的处理能力无法充分利用所有节点的带宽和算力资源，共识协议的吞吐量上限通常由领导者的数据广播能力决定。

为突破单提议者的吞吐量限制，多提议者架构允许多个节点在同一共识轮内并行提议交易。Mir-BFT\[4\]通过在多个领导者间分区请求哈希空间来防止交易重复装载，但该检查在提议时执行，属于软协调，恶意提议者仍可绕过。此外，每个领导者仍然同时承担数据传播和共识投票两项任务，没有实现数据面与控制面的分离。Narwhal\[2\]提出了数据面与共识面分离的架构，通过DAG结构实现并行装载，配合Tusk\[2\]或Bullshark\[7\]等协议完成排序，显著提升了系统吞吐量。Mysticeti\[8\]进一步采用未认证DAG降低延迟。然而，基于DAG的协议在终局机制上较为复杂，其安全性证明依赖DAG解释规则，工程实现和调试难度较高。

综上所述，单提议者协议在终局安全性和证明简洁性方面表现优异，但受限于单点吞吐瓶颈；多提议者协议提升了并行装载能力，但在终局机制和防重复保障方面仍有改进空间。更重要的是，上述协议均未考虑排序输出与执行层之间的协同关系。稳定币转账负载具有两类结构性特征：一是热点访问，即少数商户或归集账户承载大量收款写入，导致高比例写冲突；二是冲突域可预测，即Transfer类交易的读写集合近似由交易参数决定。因此，仅提高共识排序层的吞吐量并不必然提升系统端到端吞吐量——若排序输出将大量同热点域写操作在块内交织，执行层会因冲突导致级联回滚或等待，表现为吞吐塌缩与尾延迟飙升。

为此，本章提出一种执行感知的多提议者BFT共识协议MP3-BFT（Multi-Proposer Parallel Pipeline BFT）。该协议通过数据面与控制面分离以及双层证书结构，在保持链式QC可证明终局的前提下实现并行装载，并通过执行感知的确定性块内排序为并行执行提供结构化输入。本章工作的主要贡献点如下：

（1）本章设计了基于双层证书的多提议者BFT共识协议MP3-BFT。该协议采用了三维流水线设计，数据面批次传播与控制面投票在时间上重叠，同一高度内的K个槽在空间上并行，相邻高度间通过链式QC实现跨高度的流水线。理论分析表明，该协议在拜占庭容错条件下满足安全性和活性，控制面消息复杂度为$O(kn)$，共识吞吐量随并行槽数线性增长，流水线使得有效高度周期缩短至单阶段延迟。

（2）本章设计了基于投票层硬约束的桶合规（Bucket Compliance）与槽内唯一性联合反重复装载机制。桶合规将交易桶分区检查提升为验证者投票时的硬约束，从协议层面抑制跨槽重复装载；槽内唯一性约束消除同一槽内的交易重复；CADO阶段的全局去重消解跨视图重复。三层机制联合保证宏块级别的交易唯一性（引理3.1a）。理论分析表明，基于加密哈希混合的领导者选择与桶轮换机制，桶内交易等待被诚实提议者覆盖的期望视图数为$O(1)$，有效保障公平性和审查抗性，并防止重复带来的吞吐量下降。

（3）本章设计了确定性块内排序接口CADO（Conflict-Aware Deterministic Ordering）。多提议者架构下，宏块提交后各节点获得的是无序的交易集合， 而执行层需要所有节点一致的全序序列。CADO在宏块提交后根据交易携带的冲突域标识分组排序，派生出确定性的块内顺序$\pi_{h}$，为第五章的并行执行引擎提供结构化输入，形成共识层到执行层的确定性输出接口。

本章余下部分的组织结构如下：第3.2节构建了协议所需的系统模型、冲突域标识定义以及数据结构定义。第3.3节详细描述了MP3-BFT共识协议，包括数据面可用性机制、控制面双层认证与提交过程、执行感知的块内排序派生以及视图切换与容错机制。为了验证所提出的共识协议，第3.4节开展了理论分析，包括安全性证明、活性证明和复杂度分析。第3.4节将MP3-BFT与相关工作进行了结构化对比。最后，第3.6节总结了本章的研究内容。

## 3.2 模型定义

MP3-BFT是一种面向许可链的多提议者BFT共识协议。通过将交易的可用性传播与共识排序解耦为数据面和控制面，并在控制面内采用双层证书结构实现并行装载与终局提交的分离，从而提升排序吞吐量。本节将从理论层面建立MP3-BFT++协议所采用的基础模型，包括系统与安全模型、冲突域标识定义、数据面对象定义以及控制面消息与数据结构，为后续阐述具体的协议设计细节奠定理论基础。

### 3.2.1 系统与安全模型

MP3-BFT++协议的系统模型定义了参与共识的验证者集合及其故障假设。设验证者集合为$\mathcal{V}$，其规模为$n$，其中最多$f$个拜占庭故障节点，满足$n \geq 3f + 1$。拜占庭节点可能表现出任意恶意行为，包括不响应消息、发送错误消息、双重投票等。

MP3-BFT++协议采用部分同步网络模型\[1\]。该模型假设存在一个未知的全局稳定时间GST（Global Stabilization Time），在GST之前消息延迟可能无界，在GST之后消息延迟有界为$\Delta$。部分同步模型兼容了同步和异步两种网络情况，同时纳入了实际网络链路不确定性因素的考量，接近现实网络环境。在安全性方面，协议保证在任何网络条件下（包括GST之前）不会产生冲突的提交结果；在活性方面，协议保证在GST之后有限时间内系统能够持续推进提交高度。

表3.1列出了协议中使用的主要符号及其含义。

表3.1 协议符号表

| 符号                 | 含义                                                  |
|----------------------|-------------------------------------------------------|
| $$n$$                | 验证者总数                                            |
| $$f$$                | 最大拜占庭故障数，满足$n \geq 3f + 1$                 |
| $$v$$                | 视图编号，$v = 0,1,2,\ldots$                          |
| $$h$$                | 宏块高度，$h = 1,2,\ldots$                            |
| $$k$$                | 并行槽数，槽编号$s \in \{ 0,\ldots,k - 1\}$           |
| $$N_{b}$$            | 桶数量，用于反重复分配（注意与批次$\mathcal{B}$区分） |
| $$S_{\text{batch}}$$ | 批次大小，按交易数或字节计                            |

### 3.2.2 冲突域标识

多提议者架构下，宏块提交后各节点获得的是一个无序的交易集合。为使所有节点 派生出一致的块内执行顺序，需要一个确定性排序函数（即3.3.3节的CADO）。 本节定义CADO所需的唯一外部输入——冲突域标识。

**定义3.1（冲突域标识）** 每笔交易$T$携带一个冲突域标识$d(T)\mathcal{\in D}$， 其中$\mathcal{D}$为域标识符的全集。$d(T)$由交易参数确定性推导， 不依赖运行时状态。在稳定币场景下，Transfer交易的$d(T)$为收款方地址， Mint交易的$d(T)$为目标账户地址，以此类推。

冲突域标识的语义是：共享同一$d(T)$值的交易之间可能存在状态写冲突，而$d(T)$不同的交易之间写冲突概率极低。CADO利用此信息将同域交易在块内序列中聚集排列，使执行层可跨域并行、域内顺序执行。

需要说明的是，冲突域标识是更完整的冲突规约（Conflict Specification）概念 的简化投影。完整的冲突规约还包括交易的声明读写集近似，其形式化定义、在稳定币 场景下的实例化（各交易类型的读写集与域选择）、可静态验证性假设以及违反情形的 安全性影响分析将在第五章统一展开，因为这些内容主要服务于执行层的域感知并行 调度，而非本章的共识排序协议。在本章范围内，CADO仅使用$d(T)$这一标量值。

### 3.2.3 数据面对象定义

数据面负责交易的收集、打包和可用性传播。本节定义数据面的两个核心对象：批次和可用性证书。

**（1）批次**

批次是数据面的基本传播单元，将若干交易打包为一个整体进行传播和存储。批次的形式化定义为：

$$\mathcal{B = \langle}\text{tx\_list},\ \text{meta},\ \sigma_{p}\rangle,\quad h_{\mathcal{B}} = H\mathcal{(B)}$$

其中$\text{tx\_list}$为交易列表，$\text{meta}$为元数据，$\sigma_{p}$为批次生产者签名，$H( \cdot )$为抗碰撞哈希函数，$h_{\mathcal{B}}$为批次哈希。

**（2）可用性证书**

可用性证书（Availability Certificate，简称AvailCert）是对批次可用性的集体确认。其形式化定义为：

$$\text{AC}\mathcal{(B) = \langle}h_{\mathcal{B}},\ \Sigma_{2f + 1}(h_{\mathcal{B}})\rangle$$

其中$\Sigma_{2f + 1}$表示至少$2f + 1$个验证者对批次哈希$h_{\mathcal{B}}$的签名集合或聚合签名。其语义为：该批次已被足够多节点确认存储并可取回。

表3.4列出了数据面对象及其验证条件。

表3.4 数据面对象与验证条件

| 对象                    | 验证条件                                            |
|-------------------------|-----------------------------------------------------|
| 批次（Batch）           | 签名有效、哈希一致、大小$\leq S_{\text{batch}}$     |
| 可用性证书（AvailCert） | 阈值签名验证通过、指向有效批次哈希$h_{\mathcal{B}}$ |

## 3.3 MP3-BFT++共识协议

本节介绍MP3-BFT++协议的设计细节。BFT 共识的排序流程可抽象为四项功能：交易数据传播、交易集合的局部认证、局部认证的汇聚与终局化、以及确定性的块内顺序输出。在PoV等单提议者链式协议中，上述功能高度集中于领导者角色：领导者既承担带宽密集的数据广播，又负责延迟敏感的投票驱动与终局推进，并通过打包顺序隐式决定块内排序。该耦合结构使领导者出站带宽与计算负载成为系统吞吐的主要瓶颈。

为缓解数据传播瓶颈，Narwhal\[2\]将数据传播从共识控制路径中分离，通过所有节点并行广播并以 DAG 规则在本地导出全序，从而提升带宽利用率。然而，DAG 解释与链式协议的证明范式存在差异，且轮次推进对足够多顶点到达具有依赖性，系统性能可能受慢节点影响。

MP3-BFT++在继承数据面分离思想的同时，控制面仍采用链式 QC（Quorum Certificate）框架以保持证明结构的可归约性。为此，协议将排序过程组织为四个相对独立的功能层，如图3.2所示。

1.  **数据面**：将交易打包为批次并通过$2f + 1$个验证者签名形成可用性证书（AvailCert），以可用性证书（AvailCert）刻画批次数据的可用性与持久存储；

2.  **槽级认证层**：在每个高度设置$k$个并行槽，对块片提议形成槽级证书（SlotQC）；

3.  **宏块链终局层**：汇聚$k$个 SlotQC 构造宏块头，并以 MacroQC 与链式提交实现确定性终局；

4.  **执行派生接口（CADO）**：在宏块提交后，本地确定性派生块内全序$\pi_{h}$​，以增强对执行层的友好性。

该分层设计的关键在于：

1.  数据传播与终局控制解耦：支持数据面异步预取

2.  交易挑选与认证并行化：以槽为单位分摊提议与投票收集压力

3.  排序输出与共识通信解耦：CADO 为纯本地确定性计算。

下文将分别说明：数据面的批次传播与可用性证书机制（3.3.1节）、控制面的反重复分配与双层认证提交（3.3.2节）、确定性块内排序接口 CADO（3.3.3节）以及视图切换与容错机制（3.3.4节）。

### 3.3.1 数据面

数据面的核心任务是将交易集合组织为批次并在全网传播，进而形成可用性证书，为控制面提供批次内容已被足够多节点持有的可验证凭据。

数据面的具体工作流程如下：

**步骤1（批次构造）**：任意验证者从本地交易池选取交易集合，按照批次大小上限 $S_{bat}$ 生成批次$B$（定义见3.2.3节式(3.1)），计算批次哈希$h_{B} = H(B)$，并附加批次生产者签名。

**步骤2（批次广播与存储）**：批次生产者将$B$广播至全体验证者。接收方验证生产者签名及哈希一致性；验证通过后，将批次内容存储于本地，并对$h_{B}$​ 生成签名回传给生产者。

**步骤3（可用性证书生成）**：批次生产者收集不少于 $2f + 1$ 个验证者对 $h_{B}$​的签名，并按式(3.2)聚合形成可用性证书$AC(B)$。生成的$AC(B)$被加入全局 AvailCert 池，以供后续槽提议者引用。

需要指出的是，上述流程不依赖控制面当前视图（view）或高度（height）的推进状态。该特性使数据面能够与控制面并行运行，从而在控制面发生视图切换或短暂停顿时，仍可持续生成可用性证书并累积至 AvailCert 池。

**可用性证书的可取回性保障**

可用性证书$AC(B)$的核心语义在于：一旦证书有效，则批次内容在部分同步网络的 GST 之后可被诚实节点以有界延迟取回。

**引理3.1（可取回性）** 若$\text{AC}\mathcal{(B)}$有效，则至少存在$f + 1$个诚实节点持有批次内容$\mathcal{B}$。在GST之后，其他诚实节点可在$O(\Delta)$时间内取回$\mathcal{B}$。

*证明*：签名集合规模$\geq 2f + 1$，最多$f$个拜占庭节点，故至少$f + 1$个诚实签名者持有$\mathcal{B}$。在部分同步模型下，GST之后网络延迟有界为$\Delta$。诚实节点向签名者集合请求$\mathcal{B}$时，最多$f$个拜占庭节点可能拒绝或返回错误数据，但至少$f + 1$个诚实签名者将在$\Delta$时间内响应。哈希校验$H\mathcal{(B) =}h_{\mathcal{B}}$排除篡改，确保返回内容正确。在GST之前，取回可能需要无界时间，但安全性不依赖于取回完成。$▫$

### 3.3.2 控制面

控制面负责在数据面产出的AvailCert之上完成交易的挑选、认证和终局提交。控制面分为槽级认证层和宏块链终局层两级，将"并行复杂性"限制在槽级认证层，将"全局一致性与终局"集中在宏块链层，便于安全性证明与工程调试。

**（0）反重复分配：桶合规与槽内去重**

$k$个并行槽各自独立挑选AvailCert提议块片，但如果不同槽装载了相同交易，会造成带宽和执行资源浪费，甚至被恶意客户端放大为拒绝服务攻击。桶合规（Bucket Compliance）与槽内唯一性是MP3-BFT++在验证者投票时执行的两项硬约束，从协议层面抑制重复装载。这两项检查在逻辑上属于槽级认证的投票前置条件——验证者在对块片提议（BP）投SlotVote之前必须通过这些检查，不满足则拒绝投票，使不合规提议无法形成SlotQC。

需要强调的是，桶合规的设计目标是抑制跨槽重复装载而非提供全局去重保证，完整的去重语义由多层机制联合实现。

**定义3.2（交易桶映射）** 对于交易$T$，其桶映射定义为：

$$b(T) = H\left( \text{key}(T) \right)\mspace{6mu} mod\ N_{b}$$

其中推荐$\text{key}(T) = \left( \text{sender},\text{nonce} \right)$，既便于去重，也天然兼容发送者序约束。

**定义3.3（桶合规）** 设视图$v$下槽$s$的桶分配为$\text{Bkt}(v,s)$，批次$\mathcal{B}$中包含的交易集合为$\text{Tx}\left( \mathcal{B} \right)$。块片提议$\text{BP} = \left( v,h,s,\text{cert\_set},\ldots \right)$满足桶合规当且仅当：

$$\forall\text{AC} \in \text{cert\_set},\ \forall T \in \text{Tx}\left( \text{Batch}\left( \text{AC} \right) \right):b(T) \in \text{Bkt}(v,s)$$

验证者在投票前必须检查此条件，不满足则拒绝投票。

桶合规保证了视图$v$内不同槽所装载的交易桶范围互不重叠，从而从协议层面抑制跨槽重复装载。然而，桶合规本身并不禁止以下两类重复：同一槽内部，提议者可能在cert_set中重复引用包含同一交易的不同批次；跨视图或跨高度之间，由于桶分配轮换，同一交易可能在不同视图被不同槽装载。

针对第一类重复，MP3-BFT++在验证者投票检查中增加槽内唯一性约束。

**定义3.3a（槽内交易唯一性）** 验证者在对块片提议投票前，除桶合规外还需检查：提议中引用的所有批次解包后的交易集合中，不存在重复的$\left( \text{sender},\text{nonce} \right)$对。即：

$$\forall T_{i},T_{j} \in \bigcup_{\text{AC} \in \text{cert\_set}}\text{Tx}\left( \text{Batch}\left( \text{AC} \right) \right):\ i \neq j \Rightarrow \text{key}\left( T_{i} \right) \neq \text{key}\left( T_{j} \right)$$

违反此条件的提议同样被拒绝投票。由于$\text{key}(T) = \left( \text{sender},\text{nonce} \right)$的唯一性隐含了交易不可重复提交的语义（同一发送者的同一nonce只能对应一笔有效交易），该检查同时排除了同一交易被重复打包和同一sender-nonce对应不同交易内容的两种情况。

针对第二类重复（跨视图重复装载），MP3-BFT++将其视为可容忍的活性服务机制而非需要消除的错误。当视图切换发生时，前一视图中未能形成SlotQC的交易需要在新视图中被重新装载以保障活性。这些跨视图重复交易在宏块提交后的CADO排序派生阶段被消解：CADO在组装交易集合$\mathcal{T}_{h}$时，按$\left( \text{sender},\text{nonce} \right)$执行全局去重，确保最终排序序列$\pi_{h}$中不含重复交易。

以下引理总结了上述多层机制的联合去重保证。

**引理3.1a（宏块内交易唯一性）** 在桶合规（定义3.3）、槽内唯一性（定义3.3a）和CADO去重的联合作用下，任意已提交宏块的最终排序序列$\pi_{h}$中不含重复交易。

*证明*：桶合规保证视图内不同槽的交易桶范围互斥，因此跨槽重复仅可能发生在桶边界的哈希碰撞情况下（概率可忽略）。槽内唯一性保证单槽内不含重复$\left( \text{sender},\text{nonce} \right)$。对于跨视图重复（视图切换导致同一交易在多个视图被装载），CADO在派生$\pi_{h}$前按$\left( \text{sender},\text{nonce} \right)$执行全局去重。三层机制联合保证$\pi_{h}$中每个$\left( \text{sender},\text{nonce} \right)$对至多出现一次。$▫$

如图3.3所示，哈希空间被划分为$N_{b}$个桶，并按视图轮换分配给$k$个槽。验证者在投票时执行桶合规与槽内唯一性检查，以硬约束抑制多提议者重复装载，并通过轮换提供抗审查覆盖。

**公平性与审查抗性分析**

桶分配随视图轮换，避免某桶长期由恶意提议者控制。为使抗审查论证具备形式化基础，本节明确对手模型与领导者选择假设，然后给出概率界。

**对手模型**：拜占庭对手至多控制$f < n/3$个验证者节点。对手可选择哪些节点为拜占庭节点（静态腐败模型），但不能预知未来视图的领导者分配结果。对手控制的槽提议者可以选择性地不装载某些交易（审查攻击），但不能违反桶合规（否则无法形成SlotQC）。

**领导者选择假设**：MP3-BFT++采用基于视图编号的确定性轮转与加密哈希混合的槽提议者选择机制。具体地，视图$v$下槽$s$的提议者为：

$$\text{Proposer}(v,s)\mathcal{= V}\left\lbrack H\left( v \parallel s \parallel \text{epoch\_seed} \right)\mspace{6mu} mod\ n \right\rbrack$$

其中$\text{epoch\_seed}$为epoch级别的公共随机种子（由上一epoch的阈值签名聚合产生）。该机制保证：在对手不能预知$\text{epoch\_seed}$的前提下，每个槽在每个视图中分配到诚实提议者的概率为$p = (n - f)/n \geq 2/3$，且不同视图间的分配近似独立。

基于上述模型，设某桶$b$在视图$v$中被分配给槽$s$，该槽提议者为诚实节点的概率为$p$。桶$b$中的交易在连续$t$个视图内均遇到拜占庭提议者（从而被持续审查）的概率为：

$$\Pr\left\lbrack \text{持续审查}\text{t}\text{个视图} \right\rbrack = (1 - p)^{t} \leq (1/3)^{t}$$

因此交易在$t$个视图内至少一次被诚实提议者装载的概率为$1 - (1 - p)^{t}$。当$t = 10$时，该概率超过$1 - (1/3)^{10} > 1 - 1.7 \times 10^{- 5}$。

**引理3.1b（抗审查等待界）** 在上述对手模型与领导者选择假设下，桶$b$中任一交易等待被诚实提议者覆盖的视图数$W$满足$\mathbb{E}\lbrack W\rbrack \leq 1/p \leq 3/2$，且$\Pr\lbrack W > t\rbrack \leq (1/3)^{t}$。

*证明*：$W$服从参数为$p$的几何分布（每个视图独立地以概率$p$成功），因此$\mathbb{E}\lbrack W\rbrack = 1/p$。尾概率$\Pr\lbrack W > t\rbrack = (1 - p)^{t} \leq (1/3)^{t}$，呈指数衰减。$▫$

需要说明的是，上述分析依赖于视图间分配的独立性假设。当领导者选择为纯确定性轮转（无随机种子）时，对手可能提前计算出连续多个视图中某桶始终由拜占庭节点负责的恶劣排列。在此情形下，上界退化为确定性最坏情况：至多需要$n$个视图后桶完成一轮完整轮转，交易必然被诚实节点覆盖。采用加密哈希混合机制正是为了避免此类确定性攻击，将最坏情况约束提升为高概率界。

**桶空间拥塞攻击防御**：对手可能通过构造大量交易集中到特定桶，造成该桶所在槽的提议者拥塞。由于桶映射$b(T) = H\left( \text{sender},\text{nonce} \right)\mspace{6mu} mod\ N_{b}$依赖发送者地址与nonce，对手要集中攻击某个桶，需要控制大量不同的$\left( \text{sender},\text{nonce} \right)$对使其哈希值落入目标桶。在许可链环境下，发送者地址经过准入控制，对手可控地址数量有限。设对手控制$A$个地址，每地址可生成的nonce范围为$\lbrack 0,N_{\text{nonce}})$，则对手最多可向单个桶注入$A \cdot N_{\text{nonce}}/N_{b}$笔交易。当$N_{b}$足够大时（推荐$N_{b} \geq 10k$），每个桶的对手可注入量被有效稀释。此外，批次大小上限$S_{\text{batch}}$和每槽引用上限$m_{\max}$提供了自然的速率限制，进一步约束了拥塞攻击的影响范围。

**（1）槽级认证（SlotQC形成）**

槽级认证是MP3-BFT++实现并行装载的核心机制。其设计动机是：数据面产出大量AvailCert后，需要有人决定"哪些AvailCert被纳入当前高度的共识"。若由宏块领导者一人完成此项挑选并收集投票，其工作量与单提议者无异。MP3-BFT++将这项工作分摊给$k$个并行槽——每个宏块高度$h$下有$k$个槽，每槽的提议者仅负责自己桶范围内的AvailCert挑选，验证者对每个槽独立投票，由该槽收集者聚合$2f + 1$投票形成SlotQC。SlotQC的本质是一张认证书：向宏块领导者证明"这部分交易的挑选和合规验证已被多数验证者确认"。为避免聚合热点，槽收集者按槽分散，局部故障仅影响局部槽吞吐。

槽级认证的具体过程分为提议和投票两个步骤。在提议步骤中，槽提议者从本地交易池中选取属于其桶分配范围内的交易，构造批次并收集可用性证书，最终组装块片提议并广播。在投票步骤中，验证者收到块片提议后，依次验证签名、父QC、桶合规和可用性证书的有效性，验证通过后向该槽收集者发送槽投票。收集者聚合$2f + 1$个有效槽投票后形成SlotQC并广播。

**（2）宏块链终局（MacroQC形成与链式提交）**

槽级产出的$k$个SlotQC各自只能证明"某个槽的块片被多数认可"，但$k$个槽之间的相对顺序和跨高度的终局安全性需要一个统一的全局裁决。宏块链终局层承担这一角色：宏块领导者在收集窗口$\delta_{\text{slot}}$内收集尽可能多的SlotQC，并组装宏块头。注意宏块领导者此时只处理$k$个SlotQC证书（每个约200字节的聚合签名），不接触任何交易数据，是一个轻量的汇聚角色。宏块头包含各槽的SlotQC条目、父宏块QC以及块内排序派生规则版本号。宏块领导者将宏块头广播给所有验证者，验证者验证通过后发送宏块投票。领导者聚合$2f + 1$个有效宏块投票后形成MacroQC。

### 3.3.3 执行感知的确定性块内排序

宏块提交后，各节点面临一个多提议者架构特有的问题：宏块头的entries引用了$k$个槽的多批次交易，但共识层只对"包含哪些交易"达成了一致，并未规定这些交易的块内执行顺序。若各节点自行决定顺序，将导致状态分叉。因此，需要一个确定性函数将交易集合$\mathcal{T}_{h}$映射为所有节点一致的全序序列$\pi_{h}$。

最简单的方案是按交易哈希排序，这能满足确定性但对执行层不友好——高冲突交易被随机分散到序列各处，执行引擎无法利用局部性进行并行优化。CADO（Conflict-Aware Deterministic Ordering）的核心思想是：在保证确定性的前提下，利用交易携带的冲突域标识$d(T)$（定义3.1）按域分组，域内按$\left( \text{sender},\text{nonce} \right)$排序。这样，同域交易在序列中相邻排列，执行层可跨域并行、域内顺序执行，显著减少冲突导致的回滚开销。该思想与Anjana等人\[5\]提出的冲突规约一致，但将其从执行层提升至排序输出端应用。

需要强调的是，CADO不是共识协议的一部分——它不涉及任何网络通信或投票，而是宏块提交后各节点独立执行的纯本地确定性计算。其正确性保证来自两点：第一，输入一致性（由MacroQC保证所有诚实节点获得相同的交易集合）；第二，算法确定性（相同输入必然产生相同输出）。将CADO独立为接口层还允许通过ordering_rule_id对排序规则进行版本化管理，未来升级排序策略时无需修改共识协议。

Algorithm 3.6展示了CADO确定性块内排序的伪代码。

**Algorithm 3.6：CADO（冲突感知确定性块内排序）**

    输入：已提交宏块h的交易集合T_h
          每笔交易T携带: conflict_domain d(T), sender, nonce, tx_hash
    输出：确定性全序π_h
    
    1:  groups ← map: domain → list(tx)
    2:  for T in T_h do
    3:      groups[d(T)].append(T)
    4:  end for
    5:  dom_list ← sort(domains(groups)) by H(domain_id)    // 确定性域排序
    6:  π ← empty list
    7:  for dom in dom_list do
    8:      L ← groups[dom]
    9:      L1 ← stable_sort(L) by (sender, nonce)    // 发送者序
    10:     L2 ← stable_sort(L1) by tx_hash            // 确定性仲裁
    11:     π.extend(L2)
    12: end for
    13: return π

**定理3.1（CADO确定性）** 对于相同的交易集合$\mathcal{T}_{h}$和相同的ordering_rule_id，所有诚实副本运行Algorithm 3.6得到的序列$\pi_{h}$完全一致。

*证明*：CADO的确定性来自以下不变量。第一，域分组：每笔交易的冲突域$d(T)$由其冲突域标识唯一确定。第二，域排序：使用域标识符的哈希$H\left( \text{domain\_id} \right)$排序，哈希函数确定性。第三，组内排序：stable_sort按$\left( \text{sender},\text{nonce} \right)$主键、$\text{tx\_hash}$次键排序，均为确定性值。由于所有副本接收相同的已提交交易集合$\mathcal{T}_{h}$（由MacroQC保证）并使用相同算法，$\pi_{h}$在所有诚实副本上完全一致。$▫$

## 3.4 理论分析

本节从理论层面对所提出的MP3-BFT++排序协议进行深入评估，主要分析其在安全性、活性和复杂度方面的表现。

### 3.3.1 安全性

安全性分析旨在证明MP3-BFT++协议不会产生冲突的提交结果。首先给出预备定义，然后建立投票规则和提交规则，最后完成安全性定理的证明。

**（1）预备定义**

对任意宏块头$B$，定义$\text{height}(B)$和$\text{view}(B)$分别为块的高度与视图，$\text{parent}(B)$为其父宏块，$\text{id}(B) = H(B)$为块头摘要。

**定义3.6（法定人数）** 大小至少$2f + 1$的验证者集合称为法定人数（Quorum）。任意两个Quorum的交集大小至少为$f + 1$，其中至少一个为诚实节点。

**定义3.7（认证）** 若存在有效MacroQC(B)（含$\geq 2f + 1$票的聚合签名），称宏块头$B$被认证（Certified）。

**定义3.8（冲突）** 若$\text{height}(B) = \text{height}(B^{'})$且$\text{id}(B) \neq \text{id}(B^{'})$，称$B$与$B^{'}$冲突，记为$B \parallel B^{'}$。

**（2）投票规则、锁规则与提交规则**

每个验证者维护两个状态变量：highQC为已知最高认证宏块的QC，lockedQC为当前锁定QC，锁的高度记为$\mathcal{l}$。

**规则A（锁扩展投票）**\[3\]：验证者仅对满足以下两个条件的宏块头$B$投票：（a）parentQC有效；（b）$\text{height}(B.\text{parentQC}.\text{block}) \geq \text{height}(\text{lockedQC}.\text{block})$。

**规则B（锁更新）**：当观察到更高高度的有效MacroQC时，更新lockedQC为最高者。

**规则C（3-chain commit）**：若存在连续链$B_{0} \leftarrow B_{1} \leftarrow B_{2}$且$B_{1}$、$B_{2}$均被认证，则提交$B_{0}$。图3.8展示了3-chain commit规则的工作过程。

**（3）槽级与宏块级唯一性**

**引理3.2（同一槽唯一SlotQC）** 固定$(v,h,s)$，不存在两个不同块片摘要同时获得有效SlotQC。

*证明*：假设存在两个不同摘要$d \neq d^{'}$的SlotQC，各需$\geq 2f + 1$票，两个投票集合交集$\geq f + 1$，包含至少一个诚实节点对同一$(v,h,s)$进行双重投票，与诚实节点行为矛盾。$▫$

**引理3.3（同一视图高度唯一MacroQC）** 固定$(v,h)$，不存在两个不同宏块头同时获得有效MacroQC。

*证明*：同引理3.2，Quorum交集导致至少一个诚实节点对同一$(v,h)$进行双重投票，矛盾。$▫$

**（4）安全性定理**

                              │    至少1个诚实节点 u ∈ Q ∩ Q'
                              │
                ┌──────┐   ┌──┴───┐   ┌──────┐
                │ B'_h │←──│C'_{h+1}│←──│D'_{h+2}│
    链2:        └──────┘   └──────┘   └──────┘
                              │ MacroQC(C'_{h+1})
                              │ 投票集合 Q' (≥2f+1)
    
    矛盾推导:
      u 对 C_{h+1} 投票 → u 的 lockedQC 高度 ≥ h+1
      u 对 C'_{h+1} 投票 → 要求 C'_{h+1}.parentQC.height ≥ lockedQC.height
                          → 要求 height(B'_h) ≥ h+1
                          → 要求 h ≥ h+1  ← 矛盾! ✗

根据3-chain commit，提交$B_{h}$意味着存在认证链$B_{h} \leftarrow C_{h + 1} \leftarrow D_{h + 2}$，且$C_{h + 1}$、$D_{h + 2}$均被认证。同理，提交$B_{h}^{'}$意味着存在认证链$B_{h}^{'} \leftarrow C_{h + 1}^{'} \leftarrow D_{h + 2}^{'}$。

由于$B_{h} \parallel B_{h}^{'}$，两条链在高度$h$分叉。$C_{h + 1}$与$C_{h + 1}^{'}$不可能相等（否则其父必须唯一），因此$C_{h + 1} \parallel C_{h + 1}^{'}$。

令形成MacroQC($C_{h + 1}$)的投票者集合为$Q$，形成MacroQC($C_{h + 1}^{'}$)的集合为$Q^{'}$。二者均为Quorum，故$|Q \cap Q^{'}| \geq f + 1$。取交集中的任一诚实节点$u$。

由于$u$对$C_{h + 1}$投票且该块被认证，根据锁更新规则，$u$的lockedQC高度最终至少为$h + 1$。另一方面，$u$也对$C_{h + 1}^{'}$投票。按锁扩展投票规则，$u$投票$C_{h + 1}^{'}$必须满足$\text{height}(C_{h + 1}^{'}.\text{parentQC}.\text{block}) \geq \text{height}(\text{lockedQC}.\text{block})$。但$C_{h + 1}^{'}.\text{parent} = B_{h}^{'}$，其高度为$h$，而右侧至少为$h + 1$，即要求$h \geq h + 1$，产生矛盾。

因此假设导致诚实节点违反锁扩展投票规则，矛盾。故不可能同时提交同高度冲突宏块。$▫$

### 3.3.2 活性

（1）视图切换安全性

**引理3.4（最高highQC选择的锁兼容性）** 新视图领导者收集$\geq 2f + 1$条NEW_VIEW并选择最高高度的highQC作为父QC $\text{QC}^{*}$。则至少有$f + 1$个诚实节点对以$\text{QC}^{*}$为父的提案不会因锁扩展规则而拒绝投票。

*证明*：NEW_VIEW来自Quorum，其中至少$f + 1$个诚实节点。对诚实节点而言，其highQC的高度不低于其锁定高度（锁定必然对应认证证据），而$\text{QC}^{*}$取最高highQC，因此与多数诚实锁兼容。$▫$

**推论3.1（视图切换安全性）** 即使新领导者为拜占庭节点，试图以低于多数诚实锁高度的父QC构造分支，诚实节点将拒绝投票，无法形成MacroQC，因此视图切换不破坏安全性。$▫$

**（2）活性定理**

**定理3.3（GST后活性）** 在GST之后，若网络延迟有界，MP3-BFT++将以高概率持续形成MacroQC并推进提交高度。

*证明*：

*视图同步*：设GST之后消息延迟上界为$\Delta$。采用指数退避超时策略$T_{v + 1} = min(T_{\max},\rho T_{v})$，其中$\rho > 1$。根据部分同步模型，存在视图$v^{*}$使得$T_{v^{*}} > 2\Delta$且超时足够大以容纳完整协议轮次。在视图$v^{*}$之后，所有诚实节点将在$O(\Delta)$时间内收到相同的NEW_VIEW消息集合，实现视图同步。

*锁兼容性*（引理3.4）：新领导者收集$\geq 2f + 1$条NEW_VIEW，选择最高highQC作为父$\text{QC}^{*}$，与多数诚实锁兼容。

*MacroQC形成*：诚实领导者构造锁兼容的宏块头，广播给所有验证者。在GST后$\Delta$时间内，所有诚实节点收到宏块头并验证通过。由于锁兼容，$\geq 2f + 1$节点将投票，形成MacroQC。

*槽独立性*：即使部分槽无法形成SlotQC，宏块领导者仍可用其余槽entries组装宏块头。设有效槽比例为$\alpha \in (0,1\rbrack$，则吞吐按$\alpha$比例下降但系统活性不受影响。

*3-chain推进*：3-chain commit要求连续三个高度的宏块均被认证。MP3-BFT++采用加密哈希混合的领导者选择机制（见3.3.1节），每个视图的宏块领导者为诚实节点的概率为$p = (n - f)/n \geq 2/3$，且不同视图间近似独立。连续三个视图均为诚实领导者的概率为$p^{3} \geq 8/27 \approx 0.296$。设$W_{3}$为等待出现连续三个诚实领导者视图所需的视图数，其期望为$\mathbb{E\lbrack}W_{3}\rbrack = O(1/p^{3}) \leq 27/8 \approx 3.4$个视图。更精确地，$Pr\lbrack W_{3} > t\rbrack$呈指数衰减：

$$Pr\lbrack W_{3} > t\rbrack \leq (1 - p^{3})^{\lfloor t/3\rfloor}$$

因此在GST后，以高概率在$O(1)$个视图（与$n$无关）内即可出现连续三个诚实领导者视图并触发提交。

综上，在GST后有限时间内，系统将以压倒性概率持续提交宏块。即使在极端不利的随机序列中，期望等待时间仍为常数级别。$▫$

**注记**：若领导者选择退化为纯确定性轮转（无随机种子），对手可能将拜占庭节点安排在轮转序列中形成长串分布，使得最坏情况下需要$\Theta(n)$个视图才能出现连续三个诚实领导者。这正是MP3-BFT++采用加密哈希混合机制的动机之一：将最坏情况的$\Theta(n)$等待降低为高概率的$O(1)$等待。

### 3.3.3 复杂度分析

（1）消息复杂度

控制面开销可分解为槽层与宏块层。在槽层，每个宏块高度有$k$个槽，每槽形成SlotQC需要$O(n)$投票，总计$O(kn)$。在宏块层，每个高度形成MacroQC需要$O(n)$投票。因此，控制面总体消息复杂度为$O(kn)$，但其入站压力由$k$个槽收集者分摊，避免了单点收票热点。

**（2）排序吞吐量**

排序吞吐量近似为：

$$\text{TPS}_{\text{order}} \approx \frac{k \cdot m_{\max} \cdot \left| \mathcal{B} \right|}{\tau}$$

其中$\left| \mathcal{B} \right|$为平均批次交易数，$\tau$为有效高度周期。 在跨高度流水线模式下（3.3.2节(3)），相邻高度的槽级认证与宏块终局在时间上 重叠，因此$\tau = \tau_{\text{pipe}} = max\left( T_{\text{slot}},T_{\text{macro}} \right)$ （公式3.5a），而非无流水线时的$\tau_{\text{serial}} = T_{\text{slot}} + T_{\text{macro}}$。 分子中$k$对应空间并行的$k$个槽，$\tau$的缩短对应跨高度流水线的时间重叠， 两者共同决定了排序吞吐量的线性扩展能力。结合数据面带宽与执行上界，可得端到端上界：

$$\text{TPS}_{\text{e2e}} \leq min\left( \frac{\text{BW}_{\text{data}}}{\text{TxBytes}},\ C_{\text{exec}}(\kappa),\ \text{TPS}_{\text{order}} \right)$$

该不等式在第六章用于解释增大$k$或$m_{\max}$何时收益递减，并指导参数选择。

**（3）端到端瓶颈模型**

为刻画端到端瓶颈，本文采用如下近似：

$$\text{TPS}_{\text{e2e}} \approx min\left( \text{TPS}_{\text{order}},\ \text{TPS}_{\text{exec}}(\kappa) \right)$$

其中$\kappa$表示冲突强度（例如热点账户占比、Zipf参数等）。MP3-BFT++的设计目标不是孤立最大化$\text{TPS}_{\text{order}}$，而是通过执行感知输出提高$\text{TPS}_{\text{exec}}(\kappa)$在高冲突区间的可维持水平，从而抬高整体瓶颈。图3.10展示了排序吞吐量与执行吞吐量随参数变化的端到端瓶颈关系。

### 3.3.4 参数选择分析

MP3-BFT++协议的性能与活性高度依赖参数选择。本节给出关键参数的选择原则与分析。

**（1）并行槽数**$k$

增大$k$提升并行装载能力，但控制面投票数量随$k$线性增长。令每条SlotVote大小为$s_{v}$，高度周期为$\tau$，则单节点控制面出站带宽近似为：

$$\text{BW}_{\text{ctrl}}^{\text{out}} \approx \frac{k \cdot s_{v} + s_{mv}}{\tau}$$

其中$s_{mv}$为MacroVote大小。控制面带宽约束下，$k$的上界为：

$$k_{\max} = \left\lfloor \frac{\text{BW}_{\text{ctrl}} \cdot \tau - s_{mv}}{s_{v}} \right\rfloor$$

在典型配置（$\text{BW}_{\text{ctrl}} = 10$ Mbps，$\tau = 500$ ms，$s_{v} \approx 200$ B）下，$k_{\max} \approx 25$。工程上常取$k \in \lbrack 4,16\rbrack$以在并行收益与控制开销间取得平衡。

**（2）桶数**$N_{b}$

$N_{b}$决定桶合规的细粒度与负载均衡能力，其选择需平衡三方面因素。

第一，轮换覆盖粒度：每个视图将$N_{b}$个桶分配给$k$个槽，每槽分得$N_{b}/k$个桶。当视图轮换时，桶分配偏移$N_{b}/k$。设期望轮换覆盖因子（即桶总数与每槽桶数之比）为$R_{\text{coverage}} = N_{b}/k$。$R_{\text{coverage}}$越大，桶分配在连续视图间的重叠越小，抗审查覆盖越均匀。当$R_{\text{coverage}} \geq 10$时，每个桶在10个视图内完成一轮完整轮换，与引理3.1b的抗审查等待界相匹配。

第二，哈希均匀性：桶映射$b(T) = H(\text{key}(T))\mspace{6mu} mod\ N_{b}$要求$N_{b}$与哈希函数输出空间的交互具有良好均匀性。取$N_{b}$为2的幂使得取模运算等价于位掩码操作，既保证了密码学哈希输出的均匀分桶（避免非2幂模数引入的微小偏差），又将取模运算优化为$O(1)$的位运算，有利于验证者在投票前高效完成桶合规检查。

第三，元数据管理开销：每个桶需要维护当前视图下的槽归属映射（$O(1)$查表），总空间为$O(N_{b})$。当$N_{b}$过大时，虽然粒度更细但管理开销上升且边际收益递减（对哈希均匀分布的交易流，$R_{\text{coverage}} > 20$后负载均衡改善可忽略）。

综合上述三方面，推荐$N_{b} \geq 10k$，取$N_{b} = 2^{\lceil\log_{2}(10k)\rceil}$。例如$k = 8$时$N_{b} = 128$（$R_{\text{coverage}} = 16$），$k = 16$时$N_{b} = 256$（$R_{\text{coverage}} = 16$）。

**（3）收集窗口**$\delta_{\text{slot}}$

$\delta_{\text{slot}}$决定宏块领导者等待SlotQC的时间，存在吞吐—延迟权衡。建议基于RTT的指数加权移动平均（EWMA）估计$\widehat{\Delta}$自适应：

$$\delta_{\text{slot}} = \alpha\widehat{\Delta} + \beta$$

其中$\alpha \in \lbrack 1.5,2.5\rbrack$为安全系数，$\beta$为最小处理开销。$\widehat{\Delta}$的更新采用指数加权：${\widehat{\Delta}}_{\text{new}} = (1 - \gamma) \cdot {\widehat{\Delta}}_{\text{old}} + \gamma \cdot \text{RTT}_{\text{observed}}$，推荐$\gamma = 0.125$（与TCP RTT估计一致）。

**（4）视图超时**$T_{v}$

部分同步模型下建议指数退避抑制GST前震荡：

$$T_{v + 1} = min(T_{\max},\rho \cdot T_{v}),\quad\rho > 1$$

推荐$\rho = 1.5$，$T_{\max} = 60$ s。

**与Mir-BFT的关键差异**：Mir-BFT\[4\]通过在epoch领导者间分区请求哈希空间来防止交易重复装载，但该检查在提议时执行，属于软协调。MP3-BFT++将桶合规提升为投票时的硬约束：不合规提议无法形成SlotQC，从协议层面抑制跨槽重复装载。结合槽内唯一性硬约束与CADO阶段的全局去重，MP3-BFT++实现了宏块级别的交易唯一性保证（引理3.1a）。此外，MP3-BFT++通过ordering_rule_id显式支持CADO派生，为执行层提供结构化输入，这是Mir-BFT未涉及的方向。

**与Narwhal的关键差异**：MP3-BFT++借鉴了Narwhal\[2\]的数据面与共识面分离思想，但选择链式QC而非DAG作为终局机制。这一选择使得安全性证明更为直接（继承HotStuff\[3\]的锁规则框架），同时通过SlotQC层实现并行装载，在保留证明简洁性的同时获得吞吐收益。

**与HotStuff的关系**：MP3-BFT++可视为HotStuff\[3\]的多提议者扩展。宏块链层完全继承HotStuff的锁规则与3-chain commit，确保安全性证明的继承性；SlotQC层实现并行装载，将排序瓶颈从"单领导者聚合"迁移到"多槽并行认证"。

## 3.6 本章小结

本章提出了一种执行感知的多提议者BFT排序协议MP3-BFT++。该协议通过数据面与 控制面分离的架构设计，以及SlotQC/MacroQC的双层证书结构，使得多个并行槽能够 在同一宏块高度并行装载交易并独立认证，宏块链层通过链式QC与锁规则提供确定性 终局。在此基础上，协议采用三维流水线设计实现高吞吐：$k$个槽在空间上并行 装载（空间并行），相邻高度的槽级认证与宏块终局在时间上重叠（跨高度流水线）， 数据面异步预取为后续高度持续积累可用性证书（数据面预取），三者协同使有效 高度周期缩短至单阶段延迟，排序吞吐量相对单提议者串行协议获得约$2k$倍加速。

此外，本章设计了基于投票层硬约束的桶合规与槽内唯一性机制，通过多层协同 （桶合规抑制跨槽重复、槽内唯一性消除槽内重复、CADO全局去重消解跨视图重复） 实现宏块级别的交易唯一性保证。在排序输出端，本章引入冲突感知确定性排序CADO， 将交易的冲突结构显式纳入块内顺序派生，为并行执行引擎提供结构化输入，形成 共识—执行协同闭环。本章将冲突规约的可静态验证性作为显式信任假设（假设3.1）， 分析了其在稳定币许可链受限合约模板下的成立条件，并论证了即使该假设被部分 违反，执行层的验证回滚机制仍可保证正确性。理论分析证明了MP3-BFT++协议在 拜占庭容错条件下的安全性，并基于加密哈希混合的领导者选择机制给出了GST后 以高概率持续提交的活性保证。控制面消息复杂度为$O(kn)$。分散收集者与备份 容错机制使系统在多类局部故障下实现优雅退化。
